    38‚Üí1. Introduction
    39‚Üí1.1 Purpose of this Document
    40‚ÜíThis Product Requirements Document (PRD) defines the functional and non-functional requirements for the
    41‚Üídevelopment of an ICT Help Desk and Ticketing System for Jabatan Tenaga Manusia (JTM). It is intended to guide an
    42‚ÜíAI-assisted development workflow using z.ai (model GLM-5.2) as the code-generation agent, Supabase as the
    43‚Üíbackend-as-a-service platform, and Netlify as the deployment/hosting platform. This document serves as the single
    44‚Üísource of truth for scope, features, data model, workflows, and acceptance criteria.
    45‚Üí1.2 Background
    46‚ÜíJTM offices and training centres currently rely on ad-hoc channels (phone calls, WhatsApp, walk-ins) to report ICT-
    47‚Üírelated problems such as WiFi/network outages, software issues, computer hardware faults, PA system malfunctions,
    48‚Üíand media equipment problems. This results in untracked requests, delayed resolution, no accountability trail, and no
    49‚Üívisibility into ICT unit workload. A centralised, role-based, auditable ticketing system is required to formalise this
    50‚Üíprocess.
    51‚Üí1.3 Objectives
    52‚Üí‚Ä¢ Provide a single, authenticated channel for reporting ICT problems across defined categories.
    53‚Üí‚Ä¢ Enable transparent tracking of ticket status from issuance to resolution, with full timestamped history.
    54‚Üí‚Ä¢ Allow administrators to manage user accounts and assign tickets to the appropriate technician.
    55‚Üí‚Ä¢ Automate email notifications for assignment and resolution events, keeping all relevant parties informed.
    56‚Üí‚Ä¢ Provide a real-time, colour-coded dashboard visible to all authenticated users for at-a-glance status awareness.
    57‚Üí‚Ä¢ Establish a secure, low-cost, low-maintenance architecture suitable for a government department (Supabase +
    58‚ÜíNetlify).

   288‚Üí6. System Architecture
   289‚Üí6.1 High-Level Architecture
   290‚ÜíThe system follows a Jamstack architecture: a static/SPA frontend hosted on Netlify communicates directly with
   291‚ÜíSupabase (PostgreSQL database, Auth, Realtime, Storage, and Edge Functions) over HTTPS using the Supabase client
   292‚ÜíSDK. Business logic that must run with elevated privilege (e.g., creating auth users as Admin, sending emails) executes
   293‚Üíin Supabase Edge Functions rather than in the browser.
   294‚Üí‚Ä¢ Client Layer: Web application (responsive), scaffolded and iterated using z.ai (model GLM-5.2) as the AI
   295‚Üídevelopment agent.
   296‚Üí‚Ä¢ Backend-as-a-Service: Supabase ‚Äî PostgreSQL database, Supabase Auth (email/password), Row Level Security,
   297‚ÜíRealtime subscriptions, Storage (for optional attachments), and Edge Functions (serverless business logic).
   298‚Üí‚Ä¢ Notification Layer: Supabase Edge Function triggered by database webhook/trigger, calling a transactional email
   299‚ÜíAPI (e.g., Resend or SendGrid) for assignment and resolution emails.
   300‚Üí‚Ä¢ Hosting/Deployment: Netlify ‚Äî continuous deployment from Git repository, environment variables for
   301‚ÜíSupabase URL/anon key, HTTPS by default, CDN-backed static asset delivery.

   320‚Üí6.3 Architecture Flow ‚Äî Ticket Lifecycle
   321‚Üí1. Teacher logs in and submits a ticket via the web app; a row is inserted into the tickets table with status =
   322‚Üí'Issued'.
   323‚Üí2. Admin views the unassigned ticket queue and assigns it to a Technician; the tickets row is updated with
   324‚Üíassigned_to and an entry is added to ticket_status_history.
   325‚Üí3. A database trigger/webhook fires an Edge Function, which sends an assignment email to the technician and CCs
   326‚Üíthe original issuer.
   327‚Üí4. Technician updates status to 'In Progress' and later 'Resolved'; each change inserts a new ticket_status_history
   328‚Üírow and updates tickets.current_status.
   329‚Üí5. On the 'Resolved' transition, an Edge Function sends a resolution email to the issuer and all Admin accounts.
   330‚Üí6. All connected clients viewing the dashboard receive the updated status in real time via a Supabase Realtime
   331‚Üísubscription on the tickets table.

   446‚Üí8. Ticket Status Workflow & Colour Coding
   447‚ÜíEvery ticket progresses through a linear status workflow. The current status is always rendered with its associated
   448‚Üícolour across the ticket list, ticket detail page, and dashboard, giving all users an immediate visual read of ICT service
   449‚Üíhealth.
   450‚ÜíStatus Description
   451‚ÜíDefault status when a ticket is created. Displayed in Red. Ticket is awaiting Admin
   452‚ÜíISSUED
   453‚Üítriage/assignment to a technician.
   454‚ÜíDisplayed in Yellow/Amber. Set by the assigned Technician once work has begun
   455‚ÜíIN PROGRESS
   456‚Üíon the reported issue.
   457‚ÜíDisplayed in Green. Set by the assigned Technician once the issue is fixed. Triggers
   458‚ÜíRESOLVED
   459‚Üíresolution emails to the issuer and all Admins.
   460‚Üí8.1 State Transition Rules
   461‚Üí‚Ä¢ A ticket is created only in the 'Issued' state; there is no 'unassigned/draft' pre-state.
   462‚Üí‚Ä¢ Only an Admin can move a ticket from unassigned to assigned (assignment does not itself change
   463‚Üícurrent_status, which remains 'Issued' until the technician acts).
   464‚Üí‚Ä¢ Only the assigned Technician or an Admin can transition status: Issued ‚Üí In Progress ‚Üí Resolved.
   465‚Üí‚Ä¢ Status cannot be set backward from Resolved without Admin override (e.g., if a fault recurs, a new ticket should
   466‚Üíbe raised in Phase 1).
   467‚Üí‚Ä¢ Every transition is timestamped and attributed to the acting user in ticket_status_history.

   531‚Üítickets auth.uid(). Technicians can additionally SELECT/UPDATE rows where assigned_to =

   533‚ÜíReadable by any user who can read the parent ticket. INSERT restricted to the
   534‚Üíticket_status_history
